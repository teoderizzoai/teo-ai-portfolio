The Chunking Problem

Why standard splitters
fail in legal corpora.

Standard text splitters destroy the three things that matter in tax law: article hierarchy, cross-references between articles, and citable identifiers like Artikel 3.114, lid 2.

Hierarchy destroyed
Hoofdstuk → Afdeling → Artikel → Lid, all collapsed into flat tokens.
Cross-references severed
Article X referring to Article Z; relationship lost at chunk boundaries.
Citability eliminated
Answers must cite exact articles, not "chunk 4712". Legally useless otherwise.
"The system refuses before it guesses. That is not a limitation; it is the design." Architecture Principle #1 · Matteo De Rizzo
Our requirement
Every chunk must carry:
article number paragraph law name validity date RBAC classification parent_id
# chunk_id → "wet_ib_2001#art_3.114#lid_2" { "doc_id": "wet_ib_2001", "artikel": "3.114", "lid": "2", "classification": "public", "role_acl": ["helpdesk", "inspector"], "valid_from": "2024-01-01", "parent_id": "wet_ib_2001#art_3.114" }
Chunking Strategy

Hierarchy-Aware Parent-Child Splitting

PARENT Full Article · 200–1500 tokens
Artikel 3.114: Wet IB 2001
Stored in Postgres. Fetched at generation time for full context.
↓ auto-merging retrieval
CHILD · lid 1
60–300 tokens. Vector search hits here.
CHILD · lid 2
Retrieval precision lives here.
CHILD · lid 3
Parent auto-fetched on match.

When a child matches → retrieve parent → feed full article to LLM

LlamaIndex HierarchicalNodeParser handles parent-child linking. A custom extractor propagates RBAC metadata to every child node automatically.

class LegalMetadataExtractor(BaseExtractor): def extract(self, nodes): for node in nodes: if node.parent_node: for key in ("wet", "artikel", "role_acl"): node.metadata[key] = node.parent_node.metadata[key] # child inherits role_acl, no gap possible return nodes
Key guarantee
Every child node inherits role_acl from its parent article. There is no path by which a chunk enters the vector index without an RBAC payload.

Hybrid Retrieval

BM25 + Dense with
RRF Fusion

Two query shapes: neither retriever wins alone. Exact-reference queries ("Artikel 3.114 lid 2") go to BM25. Semantic concept queries go to dense search. RRF fuses both without an alpha weight to tune.

Exact-Reference Queries → BM25 wins
"ECLI:NL:HR:2023:123" · "Artikel 3.114 lid 2 Wet IB 2001"
Semantic Concept Queries → Dense wins
"kan een zzp'er de kosten van zijn thuiswerkplek aftrekken?"
RRF(d) = Σ 1 / (k + rankᵣ(d))
k = 60 · No α to tune · Robust across mixed workloads

Query Transformation

Why linear RAG fails:
three strategies.

Real tax questions break single retrieval. A query router selects the right strategy before any retrieval happens.

Query Decomposition
Multi-part questions split into 2–4 sub-queries via Phi-3-mini (local). Each retrieves independently; results deduplicated and jointly reranked against original.
Trigger: conjunctions + multiple distinct entities
HyDE: Hypothetical Document Embeddings
Vague/colloquial queries: LLM drafts a hypothetical answer in legal register, then we retrieve against its embedding. Legal vocabulary in the hypothetical beats user wording.
Trigger: short + colloquial; no legal identifiers
Direct Retrieval
Query already contains legal identifiers → BM25 boost + standard hybrid retrieve.

The hard rule

Semantic Cache

Why 0.97, not 0.95.

Common defaults (0.90–0.95) are dangerously low for tax law.

"Box 1 tax rate 2024" vs "Box 3 tax rate 2024"
Completely different answers ~0.93
"AOW premium rate 2023" vs "AOW premium rate 2024"
Different answers ~0.96
Threshold: 0.97 ✓
"what is the Box 1 bracket" vs "what's the tax rate for Box 1" → 0.98–0.99 → safe cache hit
Implementation note
0.97 is a starting point, not a fixed value. Once the application is live, cache hit/miss logs and user feedback should drive empirical tuning; the right threshold is corpus- and query-distribution-specific.
Cache key = embedding + salt
roles :: fiscal_year :: doc_type
TTL by document type
Legislation → 24h  ·  Case law → 1h
RBAC · Security

Mathematical pre-filtering.

Post-filtering = a data leak waiting to happen
Log lines, metrics, reranker inputs, or a single bug in the filter layer can expose classified payload.
Pre-filtering = mathematical guarantee ✓
The DB query itself returns a candidate set excluding forbidden documents. Classified data never enters the pipeline.
qdrant.query_points( collection_name="tax_corpus", query=query_vector, limit=40, query_filter=Filter( must=[ FieldCondition( key="role_acl", match=MatchAny(any=user_roles) ), # valid_from ≤ today ≤ valid_until ] ), )
Negative CI test: merge blocker
Helpdesk user · 100 FIOD queries · asserts zero FIOD chunks at any pipeline stage.

DevOps Readiness

Infrastructure-as-code, Kubernetes resource outline, HA/DR targets, and the full observability stack. Everything a DevOps team needs to begin provisioning the next day.

Infrastructure-as-Code

Terraform + Helm.

Every service is versioned and deployable without manual steps. Terraform owns cloud resources; Helm charts own Kubernetes workloads on Tier 1.

Terraform scope
ECR registry, Grafana Cloud workspace, OTel ingest endpoint, VPN gateway (Tier 1 to Tier 2), IAM roles for the CI runner.
Helm scope
Qdrant cluster, FastAPI query server, Phi-3-mini vLLM, Redis, OTel collector DaemonSet. One values-prod.yaml per environment.
GitOps flow
GitHub Actions builds, evaluates, and pushes images. ArgoCD reconciles Helm releases on the cluster. Merge to main triggers an automated deploy after all eval gates pass.

Observability Stack

Traces, Metrics, Logs.

Traces
OTel + Tempo
Every pipeline stage emits a span: auth, embed, BM25, Qdrant, RRF, rerank, grade, generate, cache-write. Backend: Grafana Tempo. Sampling: 100% when grader label is Ambiguous or Irrelevant, 10% otherwise.
Metrics
Prometheus + Grafana
Four alert rules: P95 TTFT > 1.5s (page), grader Irrelevant rate > 5% over 10 min (warn), Qdrant RAM > 85% (warn), cache miss rate > 80% sustained (info).
Logs
Loki + Promtail
Structured JSON logs from every pod, scraped by Promtail DaemonSet, stored in Loki. PII-stripped at the OTel collector before export to Tier 2. Retention: 30 days hot, 1 year cold on S3.
