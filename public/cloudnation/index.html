Cloud Architecture · Deployment Design

Where everything
actually runs.

The system is a hybrid: compute-heavy data stores and inference run on managed EU infrastructure, while all DevOps tooling lives on AWS and Azure. Every component is EU-resident. All sensitive data stays in the Netherlands, partly by design, partly because GDPR requires it anyway.

Tier 1 · Rijkswolk / On-Prem · Netherlands
Data & Inference
Qdrant 2-shard vector store, 20M+ chunks, int8-quantised, RBAC payload filtering
Elasticsearch 3-node BM25 cluster, same NL zone, exact-reference lookups
Postgres HA pair, parent chunk store, streaming replica
Redis Semantic cache, 32 GB in-memory, TTL per doc type
Phi-3-mini vLLM on A10G GPU, routing + grading, no external API calls
bge-m3 FastAPI embedding endpoint, same A10G node, 1024-dim
Tier 2 · AWS eu-central-1 / Azure NL · EU-resident
DevOps & Tooling
GPT-NL Final generation, Dutch government LLM, EU-hosted, outbound from Tier 1
GitHub Actions CI runner, eval harness, merge gates, shadow deployment automation
Grafana Cloud Dashboards and alerts: metric aggregates only, no raw document data
OpenTelemetry Trace collector, PII-stripped at Tier 1 edge before export
Locust Load testing, P95 TTFT benchmarks, scheduled weekly
Container Reg. ECR eu-central-1, all service images built and stored in EU
Data flow: what crosses the tier boundary
Tier 1 → Tier 2: Only metric aggregates and anonymised OTel spans. No document content, no query text, no user data.
Tier 1 → GPT-NL: The 6 retrieved chunks + the query. Never raw fiscal records. GPT-NL is EU-hosted, so this is GDPR-compliant regardless.
Tier 2 → Tier 1: Deployment artefacts only: Docker images pulled from ECR, CI signals triggering shadow/promote.
Side note
The on-prem tier turned out to double as a government compliance requirement (BIO2, Cbw). That was a pleasant side-effect; the architecture would look the same for a non-government client just running on GDPR grounds.
Hypothetical · Standard GDPR Only

If standard GDPR
is the only bar:
fully cloud.

The hybrid design exists because the data is sensitive and the client is a government entity requiring jurisdictional sovereignty (Rijkswolk), not just EU data residency. Strip that down to a commercial SaaS product or private-sector knowledge base where standard GDPR data-residency is the only bar to clear, and the architecture collapses into a single AWS region, fully managed, zero servers to babysit. Bedrock and OpenSearch run in eu-west-1 or eu-central-1; data stays EU-resident throughout. Here's what that looks like.

Data Layer
Storage & Retrieval
Qdrant Cloud (managed)
Replaces the bare-metal Qdrant cluster. Same HNSW + int8 quantisation, zero ops overhead. Single-region cluster in eu-west-1.
Amazon OpenSearch (Serverless)
Replaces the self-managed Elasticsearch cluster. BM25 index, auto-scales to query load, no shard tuning.
Amazon Aurora PostgreSQL
Serverless v2 for the parent chunk store. Scales to zero between ingestion bursts, read replicas on demand.
Amazon ElastiCache (Redis)
Managed semantic cache. Cluster mode, Multi-AZ, same TTL-per-doctype strategy as the on-prem version.
Inference Layer
Models & Embeddings
Amazon Bedrock: Claude 3.5 Sonnet
Final generation. Replaces both GPT-NL and the local Phi-3-mini. No GPU nodes to manage, pay-per-token, streaming supported.
Amazon Bedrock: Titan Embeddings v2
Replaces the self-hosted bge-m3. 1024-dim, same cosine similarity, no inference server to run. Batch ingestion via Bedrock async jobs.
Phi-3-mini (local vLLM, required)
Query decomposition, routing, and context grading. Cannot be replaced by a cloud API: these calls see the retrieved fiscal chunks, which must never leave the on-premises perimeter. Data residency is a hard constraint.
SageMaker: bge-reranker-v2-m3
Cross-encoder reranker on a serverless SageMaker endpoint. Autoscales to zero between load bursts, same 40→6 candidate logic.
Platform Layer
Compute & DevOps
Amazon ECS on Fargate
The FastAPI query-serving layer runs as a Fargate service. No EC2 to manage, scales out on p95 latency alarm. ALB in front.
AWS Step Functions: Ingestion
Replaces the bespoke ingestion pipeline. Each document fans out: parse → chunk → embed → write, fully orchestrated, retries built-in.
GitHub Actions + ECR
Same CI/CD setup. Eval gates run as Actions jobs, push images to ECR, deploy via ECS rolling update or canary.
AWS CloudWatch + X-Ray
Replaces the self-hosted OTel collector + Grafana. X-Ray traces per pipeline stage, CloudWatch dashboards for latency and grader distribution.
What changes
→Zero GPU nodes to provision or maintain. Bedrock and SageMaker Serverless handle all inference.
→No Qdrant cluster ops, no Elasticsearch shard management, no Postgres HA tuning.
→Ingestion pipeline becomes a Step Functions state machine: retries, fan-out, and dead-letter queues for free.
→Cost model flips from fixed CapEx (servers, GPU nodes) to variable OpEx (token usage, request count).
→Full-stack cold start from scratch takes hours, not days.
What stays exactly the same
→The entire retrieval pipeline: BM25 + dense, RRF fusion, cross-encoder reranking, parent-child merge.
→The LangGraph CRAG loop: grade → rewrite → retry → refuse. Logic is model-agnostic.
→The RBAC pre-filter. Qdrant Cloud supports payload filtering identically to self-hosted.
→The semantic cache threshold (0.97), TTL strategy, and cache key salting.
→All 8 CI/CD eval gates and the 48-hour shadow deployment protocol.

Our Approach

Three rules that
drive every
decision.

01 / 04

Grounding Beats Fluency

Every generated claim must be traceable to an exact source span. When in doubt, the system refuses to answer rather than producing a confident-sounding guess.

02 / 04

Security Is Mathematical

RBAC is enforced as a filter on the retrieval candidate set: not post-hoc redaction. Data a user is not cleared for never enters their context window. Full stop.

03 / 04

Retrieval Sets the Ceiling

The LLM is the smallest part of the problem. Most of our engineering effort goes into chunking, hybrid retrieval, and reranking: because you cannot generate what you did not retrieve.

04 / 04

Deployment Is Architecture

Where each service runs is a first-class design decision, not an afterthought. Data stores and inference on EU-resident managed infrastructure. DevOps tooling on cloud. The boundary is explicit and enforced.
